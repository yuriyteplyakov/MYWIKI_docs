# API —  (Application Programming Interface, программный интерфейс приложения)

> Это набор правил и способов, позволяющих разным компьютерным программам обмениваться данными и взаимодействовать друг с другом. Что-то вроде «переводчика», который позволяет одному сервису использовать функции другого, например, при оплате покупок на сайте картой.

**Простыми словами:**

API = набор правил как одна система может обратиться к другой

**Основные аспекты API:**

Принцип работы: Работает по схеме «запрос-ответ»: одно приложение (клиент) отправляет запрос к другому (серверу), который обрабатывает его и возвращает результат.

Пример: Когда вы ищете авиабилеты, сайт-агрегатор использует API для отправки запросов к базам данных разных авиакомпаний, чтобы показать актуальные цены на одном экране.

Применение: API повсеместно используются для интеграции CRM-систем, работы мобильных приложений с серверами, получения данных о погоде или оплаты через банк.

Преимущества: Ускорение разработки, так как не нужно создавать функции с нуля, а можно использовать готовые решения. 

API выступает в роли посредника, скрывая внутреннюю сложность одной системы от другой, обеспечивая простой обмен информацией.


# Есть несколько способов интеграций.


## 1. REST API (Representational State Transfer) - самый распространённый

> это архитектурный стиль взаимодействия между компонентами распределенного приложения, использующий протокол HTTP для обмена данными обычно в формате JSON.
> Он позволяет клиенту (например, браузеру) управлять ресурсами на сервере. 

**Основная идея:** - каждый объект имеет URL


### Основные характеристики и принципы REST API:

**Клиент-серверная архитектура:** Разделение ответственности; клиент не заботится о хранении данных, сервер — об интерфейсе.

**Отсутствие состояния (Stateless):** Сервер не хранит информацию о предыдущих запросах клиента. Каждый запрос должен содержать все данные, необходимые для его выполнения.

**Использование ресурсов:** Всё является ресурсом, который идентифицируется уникальным URI (например, /users/123).

**Кэширование:** Ответы сервера могут кэшироваться для ускорения работы.


**REST использует стандартные методы HTTP:**
- GET — получение данных.
- POST — создание данных.
- PUT/PATCH — обновление данных/частичное обновление.
- DELETE — удаление данных.

REST API широко используется для создания веб-приложений, мобильных приложений и микросервисов, обеспечивая высокую производительность, гибкость и независимость клиентской и серверной частей. 
Его главная особенность — использование принципов REST, которые делают взаимодействие логичным, простым и масштабируемым.

**пример**
система A → HTTP запрос → система B
Создание заявки: POST /tickets

## На примере curl из моего проекта

```
curl -X POST https://hmelon.ru/order/telegram/webhook/   -H "Content-Type: application/json"   -d '{"update_id":123,"message":{"text":"test"}}'

-X POST — указывает, что нужно использовать HTTP-метод POST

https://hmelon.ru/order/telegram/webhook/ — это URL-адрес, на который отправляется запрос

-H "Content-Type: application/json" — это заголовок (header) запроса. Он говорит серверу: «Я отправляю тебе данные в формате JSON»

-d '{"update_id":123,"message":{"text":"test"}}' — это тело (data) запроса.
```

## Как понять что проблема в интеграции

**Типичный сценарий:** Пользователь говорит: инциденты не создаются автоматически

*Возможные причины:*

- API не отвечает
- неверный токен
- ошибка JSON
- endpoint изменился


### Как искать проблему

**Алгоритм:**

1. проверить отправляется ли запрос
2. проверить HTTP код
3. проверить тело ответа
4. посмотреть логи


## Мой проект магазина на примере интеграции с Юкасса

**Суть:** В проекте реализована интеграция с платежной системой YooKassa через REST API.

Backend на Django создаёт платеж через API YooKassa и получает ссылку для оплаты. После оплаты YooKassa отправляет webhook уведомление на наш сервер. Backend обрабатывает webhook и обновляет статус заказа в базе данных.

### Логика

1. пользователь нажимает "Оплатить"
2. frontend отправляет запрос в Django
3. Django создаёт платеж через REST API YooKassa
4. YooKassa возвращает URL оплаты
5. пользователь оплачивает
6. YooKassa отправляет webhook
7. backend обновляет статус заказа

**сама интеграция происходит в приложении payment**

