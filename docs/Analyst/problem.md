
# Инцидент — это любое событие, которое нарушает нормальную работу сервиса.

**пример:** пользователь не может создать заявку

**Чем инцидент отличается от проблемы?**

> инцидент - что-то сломалось
> проблема - корневая причина

## Жизненный цикл инцидента

> New - пользователь создал заявку
> Assigned - назначили инженера
> In progress - идет анализ
> Resolved - решение найдено
> Closed - подтверждение пользователя

# Универсальный алгоритм анализа инцидента

- уточнить симптом
- воспроизвести проблему
- проверить интерфейс
- проверить сетевой запрос (API)
- проверить логи приложения
- проверить данные в базе данных
- проверить интеграции

**Выясняем:**
> что сломалось
> почему
> как исправить

## Анализ требований

1. интервью с заказчиком
2. описание AS IS (заявка создается, менеджер согласует, IT выполняет)
3. описание TO BE (заявка, автоматическое назначение, согласование, выполнение)
4. согласование

> Обычно процесс начинается с обсуждения проблемы клиента. Затем анализируется текущий процесс (as-is), после чего формируется желаемый процесс (to-be).
> После согласования требований готовится техническое задание для разработчиков.

### Настраиваем процессы

> инцидент → назначение → обработка → решение → закрытие

# Реальные инциденты enterprise систем

Заявка не создаётся

Возможные причины:

```
ошибка валидации
ошибка API
ошибка workflow
ошибка БД
```

## Что делать если система упала?

1. подтвердить инцидент
2. проверить логи
3. проверить инфраструктуру
4. локализовать проблему
5. предложить workaround

## пользователь говорит что система не работает — что будешь делать?

```
Сначала уточню детали проблемы:
какой модуль системы, какая операция вызывает ошибку.
Затем попробую воспроизвести проблему.
После этого проверю логи приложения и сетевые запросы.
Если проблема связана с данными — проверю записи в базе.
```

## клиент говорит что заявка не создается Что делать?

1. воспроизвести проблему (Попробовать самостоятельно создать заявку несколько раз (проверка стабильности проблемы))
2. проверить UI Есть ли ошибка (пользовательский интерфейс)
3. проверить API (Сработал ли запрос)
4. посмотреть логи (Есть ли ошибка backend)
5. проверить БД (Создалась ли запись)

## Клиент хочет новый процесс.

**заявка → согласование → закупка**

1. собрать требования
2. описать workflow
3. согласовать
4. передать разработчикам

## Клиент хочет новый процесс согласования заявок.

```
Сначала я уточню требования клиента.

Я бы провёл интервью и выяснил:

какие типы заявок требуют согласования
кто должен согласовывать
сколько уровней согласования
какие условия переходов между этапами

После этого я опишу процесс:

AS IS — как работает сейчас
TO BE — как должно работать

Далее подготовлю описание workflow и передам его команде разработки или настрою процесс в системе.
```

## клиент говорит система работает медленно

```

Проверил производительность базы данных.
Проверил наличие медленных SQL-запросов.
Проверил сетевые задержки.
Проверил работу интеграций с другими системами.

После этого можно локализовать узкое место.
```

## Пользователь сообщает:

> «Я нажимаю создать заявку, но ничего не происходит».
> Как вы будете диагностировать проблему?

```
Я бы действовал по следующему алгоритму:

1 Попробовал воспроизвести проблему.

2 Проверил, появляется ли ошибка в интерфейсе.

3 Проверил сетевой запрос к серверу (через DevTools).

4 Посмотрел логи backend-сервиса.

5 Проверил, создаётся ли запись в базе данных.

6 Если заявка не создаётся, проверил бы workflow или бизнес-логику обработки запроса.

Это позволяет определить, на каком уровне возникает проблема:

frontend
API
backend
 база данных

Если проблема воспроизводится только у одного пользователя, я бы проверил его права доступа.
```

## Вы нашли в логах запись: ERROR Database connection timeout Что это может означать?

```
Это означает, что приложение не смогло установить соединение с базой данных в течение заданного времени.

Причины могут быть:

база данных недоступна
высокая нагрузка
неправильные параметры подключения
проблемы сети

Следующим шагом я бы:

Проверил доступность базы данных
Проверил нагрузку на сервер
Проверил настройки подключения
```
