## Анализ требований

1. интервью с заказчиком
2. описание AS IS (заявка создается, менеджер согласует, IT выполняет)
3. описание TO BE (заявка, автоматическое назначение, согласование, выполнение)
4. согласование

> Обычно процесс начинается с обсуждения проблемы клиента. Затем анализируется текущий процесс (as-is), после чего формируется желаемый процесс (to-be). После согласования требований готовится техническое задание для разработчиков.

### более подробно для аналитика

**Основные этапы работы аналитика:**

1. Выявление (сбор) информации:

Интервью и встречи: Беседы с заказчиками, экспертами, пользователями (индивидуальные или групповые).

Анализ документации: Изучение текущих бизнес-процессов, инструкций, отчетов и конкурентов.

Наблюдение: Изучение реальной работы пользователей на их рабочих местах (shadowing).

Мозговые штурмы и опросы: Использование для сложных проектов с большим количеством участников.

2. Анализ и моделирование:

Обработка информации, выделение требований к функциям, интерфейсу и производительности.

Моделирование: Создание диаграмм бизнес-процессов (BPMN), Use Case диаграмм, прототипов интерфейсов (UI/UX).

Приоритизация требований (например, по методу MoSCoW).

3. Документирование:

Фиксация требований в формате спецификации (SRS — Software Requirements Specification).

Создание пользовательских историй (User Stories) или сценариев использования (Use Cases).

4. Проверка (валидация) требований:

Рецензирование: Согласование документации с заказчиками и командой разработчиков.

Проверка на непротиворечивость, однозначность, полноту и реализуемость.

5. Управление требованиями:

Работа с изменениями, фиксация версий требований, отслеживание реализации. 

**Ключевые принципы:**

Итеративность: Процесс повторяется, требования уточняются по мере погружения в проект.

Визуализация: Использование схем облегчает понимание (BPMN, UML).

Конструктивная критика: Активное привлечение команды к анализу требований.
